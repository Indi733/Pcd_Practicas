\section{Ejercicio 1 — ReentrantLock}

% ──────────────────────────────────────────
\subsection{Recursos no compartibles}

En este ejercicio, cada hilo trabaja con sus propias matrices locales, así que no comparten datos entre ellos. El único recurso no compartible es:
\begin{itemize}
  \item \textbf{La pantalla (\texttt{System.out})}: Imprimir la matriz completa requiere varias llamadas a \texttt{println} seguidas. Si los dos hilos imprimen a la vez, sus líneas se mezclan y la salida es ilegible. La pantalla es un recurso hardware que requiere acceso exclusivo para escritura.
\end{itemize}

% ──────────────────────────────────────────
\subsection{Condiciones de sincronización}

En este problema \textbf{no hay condiciones de sincronización}. Según la teoría, una condición de sincronización es una relación de precedencia entre procesos: un proceso tiene que esperar a que otro haga algo concreto antes de poder continuar. Aquí ningún hilo depende del resultado del otro para hacer su trabajo; cada uno genera su propia matriz y la imprime de forma totalmente independiente.

Lo que sí existe es \textbf{exclusión mutua} sobre la pantalla. El bloque de impresión de cada hilo es una sección crítica, y se protege con un \texttt{ReentrantLock} compartido (\texttt{micerrojo}). Cada hilo hace \texttt{lock()} antes de empezar a imprimir y \texttt{unlock()} al terminar, garantizando que ningún otro hilo pueda imprimir a la vez.

\subsection{Cuestiones}

\textbf{¿Qué ocurriría si no se utilizase sincronización?}

Sin sincronización, las líneas de los dos hilos se mezclarían en la consola. Como cada hilo hace varias llamadas a \texttt{println} seguidas, el sistema operativo puede interrumpirle entre una y otra para ejecutar el otro hilo. El resultado sería algo como esto:

\begin{lstlisting}[language={}, frame=leftline, basicstyle=\ttfamily\small, caption={Ejemplo de salida intercalada (sin cerrojo)}]
A x A
A + A
a1 a2 a3      a1 a2 a3      A2
a1 a2 a3      a1 a2 a3      2A 
a4 a5 a6   x  a4 a5 a6   =  r1 r2 r3
\end{lstlisting}

Es decir, las líneas de \texttt{HiloMult} y \texttt{HiloSuma} aparecen entremezcladas y la salida es ilegible.

\section{Ejercicio 2 — Semáforos}

\subsection{Recursos no compartibles}

Los recursos que no pueden ser usados por dos hilos a la vez en este ejercicio son:
\begin{itemize}
  \item \textbf{Los 3 paneles físicos}: Cada panel es un recurso hardware de escritura. Dos hilos no pueden escribir en el mismo panel al mismo tiempo sin que la salida se corrompa, así que cada panel individual requiere acceso exclusivo.
  \item \textbf{El array \texttt{libres}}: Es la variable compartida (recurso software) que guarda qué paneles están ocupados o libres. Si dos hilos lo leen y modifican a la vez sin protección, pueden acabar asignándose el mismo panel.
\end{itemize}

\subsection{Condiciones de sincronización}

Sí existe una condición de sincronización en este problema. Un equipo no puede simplemente ir a buscar un panel libre en cualquier momento; primero tiene que esperar a que haya al menos uno disponible. Es decir, un hilo tiene que esperar a que otro hilo (equipo) termine de usar su panel y lo libere. Eso es exactamente una condición de sincronización: un proceso espera a que el estado del sistema cambie por acción de otro proceso.

Esta condición se implementa con el semáforo general \texttt{disponibles}, inicializado a 3 (el número de paneles). Cada vez que un equipo quiere usar un panel hace \texttt{wait(disponibles)}, lo que le bloquea si ya hay 3 equipos usando paneles. Cuando termina, hace \texttt{signal(disponibles)} para avisar al siguiente que está esperando.

Además, el acceso al array \texttt{libres} para buscar y marcar qué panel se va a usar (y luego para liberarlo) se protege con \textbf{exclusión mutua} mediante el semáforo binario \texttt{mutex}, para que dos hilos no modifiquen el array a la vez.

\subsection{Pseudocódigo comentado}

\begin{lstlisting}[language={}, caption={Pseudocódigo HiloEquipo}]

                deportistas[a][0][i] = rand.nextInt(61) + 90;
                deportistas[a][1][i] = rand.nextInt(401) + 400;
                deportistas[a][2][i] = rand.nextInt(71) + 10;
                deportistas[a][3][i] = rand.nextInt(301) + 300;
                deportistas[a][4][i] = rand.nextInt(151) + 150;
            }

        // Calcular limites y pasarlos a matriz R
        for (int i = 0; i < 4; i++) {

                libres[i] = false;

        paneles[aacceder].escribir_mensaje("Usando panel " + aacceder + " equipo " + idequipo + "\n");

        paneles[aacceder].escribir_mensaje("Terminando panel " + aacceder + " equipo " + idequipo + "\n\n");

        // Exclusion mutua para actualizar disponibilidad del hueco tomado
        wait(mutex);                            // Protoc. Entr.
        libres[aacceder] = true;
        signal(mutex);                          // Protoc. Sal.

        // Avisa al siguiente que pueda estar esperando
        signal(disponibles);                    // Sincronizacion
    }
\end{lstlisting}

\subsection{Cuestiones}

\textbf{a) Si consideramos un panel solamente ¿cómo es el acceso al mismo? Si consideramos los tres paneles ¿cómo es el acceso? Explica ambas situaciones.}

\begin{itemize}
    \item \textbf{Un solo panel}: El acceso es en \textbf{exclusión mutua}. Un panel es un recurso que solo puede usar un equipo a la vez; mientras un hilo está escribiendo en él, ningún otro puede hacerlo.
    \item \textbf{Los tres paneles}: Pueden escribir hasta 3 equipos a la vez, uno en cada panel. Es decir, el acceso es concurrente: varios hilos operan simultáneamente, pero nunca dos sobre el mismo panel.
\end{itemize}

\textbf{b) ¿Qué tipo de semáforos has usado para resolver el problema (binarios, generales)?}

Se han usado los dos tipos:
\begin{itemize}
    \item \textbf{Semáforo binario (\texttt{mutex}, inicializado a 1)}: Para proteger en exclusión mutua el acceso al array \texttt{libres}. Así dos hilos no pueden leerlo y modificarlo a la vez y asignarse el mismo panel.
    \item \textbf{Semáforo general (\texttt{disponibles}, inicializado a 3)}: Para controlar cuántos equipos pueden estar usando paneles a la vez. Si los 3 paneles están ocupados, el siguiente equipo se bloquea en \texttt{wait(disponibles)} hasta que alguno haga \texttt{signal(disponibles)}.
\end{itemize}

\section{Ejercicio 3 — Monitores}

\subsection{Recursos no compartibles}

En esta simulación, los recursos que no pueden usarse por varios clientes a la vez son:
\begin{itemize}
  \item \textbf{Los 3 tornos de acceso}: Cada torno es un recurso que solo puede gestionar la entrada de un cliente a la vez. Dos clientes no pueden pasar por el mismo torno simultáneamente.
  \item \textbf{Cada zona de entrenamiento (las 4 zonas)}: En la implementación, cada zona se modela como un recurso que solo puede tener un cliente entrenando a la vez (\texttt{ocupado[zona]}). Aunque hay 5 máquinas físicas, la variable de control es booleana por zona, así que se trata como acceso exclusivo.
  \item \textbf{La bicicleta premium}: Es un recurso único dentro de la zona Cardio. Solo un cliente puede usarla a la vez (\texttt{biciPremiumOcupada}).
  \item \textbf{La pantalla (\texttt{System.out})}: Imprimir el informe de cada cliente son varias líneas seguidas. Sin protección, los mensajes de distintos clientes se mezclarían.
\end{itemize}

\subsection{Condiciones de sincronización}

Hay varias condiciones de sincronización, todas del mismo tipo: un cliente tiene que esperar a que otro cliente libere un recurso antes de poder usarlo.
\begin{itemize}
  \item \textbf{Espera para entrar por un torno}: Si los tres tornos están ocupados, el cliente tiene que quedarse bloqueado hasta que otro cliente termine y libere el suyo. El cliente espera en la cola del torno que tenga menos gente. Esta condición se gestiona con \texttt{await()} en la variable de condición del torno elegido.
  \item \textbf{Espera para entrar en una zona}: Si la zona elegida ya está ocupada, el cliente tiene que esperar hasta que el cliente que está dentro salga y haga \texttt{signal()}. Cada zona tiene su propia cola de espera (\texttt{Condition}).
  \item \textbf{Espera para usar la bicicleta premium}: Si un cliente quiere la bicicleta y ya la tiene otro, se queda bloqueado en la cola exclusiva de la bici hasta que ese cliente la libere y haga \texttt{signal()} en esa condición.
\end{itemize}

\subsection{Cuestiones}
\textbf{a) Indica si la acción de entrenar es concurrente o en exclusión mutua y justifica.}

Entrenar es \textbf{concurrente}. El entrenamiento se simula con \texttt{Thread.sleep(Y)}, y esa llamada se hace \textbf{fuera} de cualquier método del monitor, es decir, el hilo ya ha liberado el cerrojo antes de ponerse a «dormir». Como en ese momento no se toca ningún recurso compartido, varios clientes pueden estar entrenando a la vez sin ningún problema.

\textbf{b) ¿Qué tipo de monitor Java has usado? Justifica.}

Se ha implementado el monitor con \texttt{ReentrantLock} y variables \texttt{Condition}, usando \texttt{await()} para bloquear y \texttt{signal()} para despertar, que son los equivalentes Java de \texttt{delay} y \texttt{resume} del pseudocódigo. La política de desbloqueo que aplica este mecanismo es \textbf{DC (Desbloquear y Continuar)}: cuando un hilo llama a \texttt{signal()}, el hilo despertado no puede ejecutarse de inmediato, sino que queda pendiente hasta que el hilo que señalizó termine su método y libere el cerrojo. Hemos elegido esta solución porque necesitamos varias variables de condición independientes: una por cada zona de entrenamiento y otra exclusiva para la bicicleta premium, ya que cada una gestiona una cola de espera distinta.

\textbf{c) En el monitor, ¿has usado notify/signal o notifyAll/signalAll? Justifica.}

Hemos usado \textbf{\texttt{signal()}}. Al tener una variable de condición distinta para cada recurso (cada zona y la bici premium), cuando hacemos \texttt{signal()} sobre la condición de una zona concreta solo se despierta al cliente que está esperando exactamente ese recurso. Con \texttt{signalAll()} despertaríamos a todos los hilos bloqueados, la mayoría comprobaría su condición, vería que no se cumple y volvería a bloquearse, lo cual no tiene ningún sentido y es ineficiente.

\textbf{d) ¿Cómo se ha resuelto la exclusión mutua de la pantalla en este problema?}

Con un \texttt{ReentrantLock consola} compartido entre todos los clientes. Antes de imprimir, el hilo hace \texttt{consola.lock()} y al terminar \texttt{consola.unlock()} dentro de un bloque \texttt{finally} para asegurarse de que siempre se libera (aunque falle algo). De este modo solo un cliente puede imprimir a la vez y las líneas de distintos clientes no se mezclan.

package Ejercicio3;

import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

/**
 * Monitor que gestiona el acceso a los 3 tornos del gimnasio.
 */
public class MonitorTorno {
	
	private ReentrantLock cerrojo;
	private Condition[] colaTorno;
	private boolean[] ocupado;
	private int[] enCola;
	
	/**
	 * Inicializa el estado del monitor de tornos.
	 */
	public MonitorTorno() {
		this.cerrojo = new ReentrantLock();
		this.colaTorno = new Condition[] {
			cerrojo.newCondition(),
			cerrojo.newCondition(),
			cerrojo.newCondition()
		};
		this.ocupado = new boolean[] {false, false, false};
		this.enCola = new int[] {0, 0, 0};
	}
	
		
	/**
	 * Asigna un torno al cliente.
	 * Si no hay tornos libres, espera en la cola del torno con menor espera.
	 *
	 * @param id identificador del cliente
	 * @return identificador del torno asignado
	 */
	public int entrar(int id) {
	    cerrojo.lock();
	    int tornoElegido = -1;
	    try {
	        // 1. Buscar primer torno libre
	        for (int i = 0; i < 3; i++) {
	            if (!ocupado[i]) {          
	                tornoElegido = i;
	                break;
	            }
	        }

	        // 2. Si no hay libre elegir el de menor cola
	        if (tornoElegido == -1) {   	
	            tornoElegido = 0;
	            for (int i = 1; i < 3; i++) {
	                if (enCola[i] < enCola[tornoElegido]) {
	                    tornoElegido = i;
	                }
	            }
		        // 3. Esperar en la cola del torno elegido
		        enCola[tornoElegido]++;
		        while (ocupado[tornoElegido]) {
		            colaTorno[tornoElegido].await();
		        }
		        enCola[tornoElegido]--;
	        }
	        // 4. Reclamar el torno
	        ocupado[tornoElegido] = true;
	        return tornoElegido;
	        
	    } catch (InterruptedException e) {
	        throw new RuntimeException(e);
	    } finally {
	        cerrojo.unlock();
	    }
	    
	}
	
	/**
	 * Libera un torno y despierta a un cliente que espera en su cola.
	 *
	 * @param tornoid identificador del torno a liberar
	 */
	public void salir(int tornoid) {
		cerrojo.lock();
		try{
			ocupado[tornoid] = false;
			colaTorno[tornoid].signal();
		}finally {
			cerrojo.unlock();
		}

\section{Ejercicio 4 — Paso de Mensajes Asíncrono}

\subsection{Recursos no compartibles}

En este ejercicio los recursos que no se pueden usar a la vez son:
\begin{itemize}
  \item \textbf{El torno R y el torno L}: Cada torno solo puede atender a un aficionado al mismo tiempo. Si dos aficionados intentaran usar el mismo torno a la vez, habría conflicto. Son dos recursos independientes, así que sí se pueden usar en paralelo entre sí (uno en R y otro en L al mismo tiempo), pero no dos en el mismo.
  \item \textbf{La pantalla (\texttt{System.out})}: Imprimir el informe de cada aficionado son varias líneas. Sin protección, los mensajes se mezclan.
\end{itemize}

\subsection{Condiciones de sincronización}

Hay dos condiciones de sincronización en este ejercicio:
\begin{itemize}
  \item \textbf{Espera para usar el torno asignado}: Tras recibir del controlador qué torno le corresponde, el aficionado envía una petición al buzón de ese torno (\texttt{peticionTornoR} o \texttt{peticionTornoL}) y luego hace \texttt{receive()} en su buzón privado esperando confirmación. Si el torno está ocupado, el controlador no atiende ese buzón (porque la guarda está cerrada) y el aficionado se queda bloqueado hasta que el torno se libere y el controlador le responda. Un aficionado espera a que otro aficionado libere el torno antes de poder usarlo.
  \item \textbf{Exclusión mutua de la pantalla}: Un aficionado tiene que esperar a que otro termine de imprimir antes de poder imprimir él. Esto se implementa con el buzón \texttt{mutexPantalla}: al principio hay un mensaje dentro (el testigo). Para imprimir, el aficionado hace \texttt{receive(mutexPantalla)}, que le bloquea si otro ya tomó el testigo. Al terminar, hace \texttt{send(mutexPantalla)} para devolver el testigo y que el siguiente pueda pasar. Es la forma de hacer exclusión mutua con paso de mensajes.
\end{itemize}

\subsection{Cuestiones}
\textbf{a) ¿Se pueden usar simultáneamente los dos tornos? Justifica la respuesta.}

Sí. El controlador tiene dos variables independientes (\texttt{tornoRLibre} y \texttt{tornoLLibre}) y usa guardas en el \texttt{select} para cada buzón por separado. Si el torno R está ocupado, su guarda se cierra y el controlador deja de atender peticiones para R, pero la guarda de L sigue abierta. Así puede haber un aficionado en R y otro en L al mismo tiempo sin ningún problema.

\textbf{b) ¿Cómo has resuelto la exclusión mutua de la pantalla?}

Con un buzón dedicado (\texttt{mutexPantalla}) que actúa como testigo (token). Al arrancar, el controlador deposita el testigo en ese buzón. Antes de imprimir, cada aficionado hace \texttt{receive(mutexPantalla)}: si el buzón está vacío porque otro aficionado ya tomó el testigo, se bloquea hasta que ese otro haga \texttt{send(mutexPantalla)} y lo devuelva. De este modo solo un aficionado puede estar imprimiendo a la vez.

/**

public class HiloControlador extends Thread{
	private MailBox buzonPeticiones;
	private MailBox[] buzonRespuestas;
	private MailBox buzonLiberacion;
	private MailBox mutexPantalla;
	private Selector sel;
	private MailBox peticionTornoR;
	private MailBox peticionTornoL;
	private boolean tornoRLibre;
	private boolean tornoLLibre;
	private Random rand;
	private int liberacionesProcesadas;
	private static final int TOTAL_VALIDACIONES = 50 * 5;

	
	
	/**
	 * Crea el controlador con todos los buzones compartidos.
	 *
	 * @param buzonPeticiones buzon de peticiones de aficionados
	 * @param buzonRespuestas buzones de respuesta por aficionado
	 * @param buzonLiberacion buzon para liberar torno
	 * @param mutexPantalla buzon usado como mutex de pantalla
	 * @param peticionTornoR buzon de peticiones del torno R
	 * @param peticionTornoL buzon de peticiones del torno L
	 */
	public HiloControlador(MailBox buzonPeticiones, MailBox[] buzonRespuestas, MailBox buzonLiberacion,MailBox mutexPantalla, MailBox peticionTornoR,  MailBox peticionTornoL) {
		super();

		this.peticionTornoL = peticionTornoL;
		this.peticionTornoR = peticionTornoR;

		this.rand = new Random();
		this.tornoRLibre = true;
		this.tornoLLibre = true;
		this.liberacionesProcesadas = 0;


		this.sel = new Selector();
		sel.addSelectable(buzonPeticiones, false);
		sel.addSelectable(peticionTornoR, false);
		sel.addSelectable(peticionTornoL, false);
		sel.addSelectable(buzonLiberacion, false);

	 * hasta completar todas las validaciones previstas.

	@Override

        mutexPantalla.send("llavePantalla");
		while(true) {
			
		buzonPeticiones.setGuardValue(true);
		peticionTornoR.setGuardValue(tornoRLibre);
		peticionTornoL.setGuardValue(tornoLLibre);
        buzonLiberacion.setGuardValue(!tornoRLibre || !tornoLLibre);
   
			switch (sel.selectOrBlock()) {
			case 1: {
				int id = (int) buzonPeticiones.receive();
				int tEstimado = rand.nextInt(10)+1;

				if(tEstimado > 5) {

					torno = "R";

				break;
				
			}
			case 2: {
				int id = (int) peticionTornoR.receive();
				tornoRLibre = false;
				buzonRespuestas[id].send("llave");
				break;
			}
			case 3:{
				int id = (int) peticionTornoL.receive();
				tornoLLibre = false;
				buzonRespuestas[id].send("llave");
				break;
			}
				
				
		
			case 4: {
				
				String torno = (String) buzonLiberacion.receive();
				liberacionesProcesadas++;
				if(torno.equals("R")) {
					tornoRLibre = true;
			}
				else {
					tornoLLibre = true;
				}
				if (liberacionesProcesadas >= TOTAL_VALIDACIONES) {
					return;
				}
				break;
			}
			}

		}
